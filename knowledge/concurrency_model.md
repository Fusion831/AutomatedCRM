# Lemma Python SDK Concurrency Model

This document outlines the concurrency architecture of the Lemma Python SDK, explaining async execution patterns, method signatures, real-time listener behaviors, and thread-safety considerations.

---

## 1. Async Support

### High-Level SDK (`lemma_sdk.Lemma`)
The high-level wrappers in the SDK (e.g., `Lemma`, `Pod`, `PodTables`, `PodRecords`, `PodAgents`) are designed **purely around synchronous, blocking calls**. 
- They utilize a synchronous `httpx.Client` internally via `LemmaTransport`.
- Methods delegate to the autogenerated openapi-client's `.sync_detailed()` or `.sync()` methods, which block the calling thread during HTTP I/O.
- Running these methods directly inside an asynchronous loop (such as a FastAPI request handler) will block the main event loop.

### Low-Level Generated Client (`lemma_sdk.openapi_client`)
Under the hood, the autogenerated SDK layer (`openapi_client`) natively supports both synchronous and asynchronous network calls using `httpx`. For every API endpoint, it exposes both:
- `sync` / `sync_detailed`: Synchronous blocking requests.
- `asyncio` / `asyncio_detailed`: Asynchronous non-blocking requests.

## 2. Method Signatures & Concurrency Patterns

Because the high-level SDK does not expose async methods (e.g. there is no `client.tables.acreate`), developers must choose one of two patterns to achieve asynchronous execution.

### Pattern A: Offloading to Thread Pools (Recommended for High-Level SDK)
Since the high-level SDK is synchronous, calls can be run in worker threads using `asyncio.to_thread` to prevent blocking the event loop.

```python
import asyncio
from lemma_sdk import Lemma
from lemma_sdk.openapi_client.models.create_table_request import CreateTableRequest
from lemma_sdk.openapi_client.models.table_column import TableColumn
from lemma_sdk.openapi_client.models.column_type import ColumnType

async def main():
    # Initialize the synchronous SDK client
    lemma = Lemma(token="your_token", base_url="https://api.lemma.example")
    pod = lemma.pod("pod_uuid")

    # Define a table creation payload
    request = CreateTableRequest(
        table_name="orders",
        primary_key_column="id",
        columns=[
            TableColumn(name="id", type=ColumnType.TEXT, required=True),
            TableColumn(name="price", type=ColumnType.DOUBLE, required=False),
        ]
    )

    # Run the blocking call safely in a thread pool
    table = await asyncio.to_thread(pod.tables.create, request)
    print(f"Created table: {table.table_name}")

if __name__ == "__main__":
    asyncio.run(main())
```

### Pattern B: Natively Awaiting Low-Level Endpoints
If a thread pool is not desired, developers can extract the client's internal credentials and call the low-level async endpoints directly.

```python
import asyncio
from lemma_sdk import Lemma
from lemma_sdk.openapi_client.api.tables import table_create
from lemma_sdk.openapi_client.models.create_table_request import CreateTableRequest
from lemma_sdk.openapi_client.models.table_column import TableColumn
from lemma_sdk.openapi_client.models.column_type import ColumnType

async def main():
    lemma = Lemma(token="your_token", base_url="https://api.lemma.example")
    
    # 1. Access the low-level AuthenticatedClient
    # 2. Extract UUIDs needed for path args
    client = lemma.generated
    pod_id = lemma.settings.pod_id

    request = CreateTableRequest(
        table_name="customers",
        primary_key_column="email",
        columns=[
            TableColumn(name="email", type=ColumnType.TEXT, required=True),
            TableColumn(name="name", type=ColumnType.TEXT, required=False),
        ]
    )

    # Invoke the native async openapi endpoint
    table = await table_create.asyncio(
        client=client,
        pod_id=pod_id,
        body=request
    )
    print(f"Created table: {table.table_name}")

if __name__ == "__main__":
    asyncio.run(main())
```

## 3. Real-Time Event Listening (Server-Sent Events)

The SDK does **not** implement WebSockets for event consumption. Instead, it utilizes **Server-Sent Events (SSE)** via chunked HTTP streaming for real-time conversation and agent run events.

- **Mechanics**: Methods like `pod.conversations.stream(conversation_id)` and `pod.conversations.send_stream(...)` configure a request using `stream=True` and pass it to a synchronous `httpx.Client`.
- **Consumption**: The returned response is a blocking iterable. The caller must iterate over it using a loop. The thread blocks waiting for new chunks of data to arrive.
- **No Background Loop**: The SDK does not manage background threads or thread loops for streaming. Managing background processing of streams is entirely the developer's responsibility.

### Stream Consumption Blueprint (Sync vs Async)

#### Synchronous Stream Consumer
```python
from lemma_sdk import Lemma

lemma = Lemma(token="your_token", base_url="https://api.lemma.example")
pod = lemma.pod("pod_uuid")

# Run agent and open a stream
response = pod.agents.run(
    name_or_id="support_agent",
    message="Generate an invoice for order 12",
    stream=True
)

try:
    # Iterate over stream blocks; this loop blocks the thread until chunks arrive
    for line in response.iter_lines():
        if line.startswith("data:"):
            # Process Server-Sent Event data
            print(line)
finally:
    response.close()
```

#### Asynchronous Stream Consumer (FastAPI/asyncio Integration)
```python
import asyncio
from lemma_sdk import Lemma

async def consume_agent_stream(pod, conversation_id):
    # Offload the stream generation and iteration to a thread pool
    def read_stream():
        response = pod.conversations.stream(conversation_id)
        events = []
        try:
            for line in response.iter_lines():
                if line.startswith("data:"):
                    events.append(line)
            return events
        finally:
            response.close()

    events = await asyncio.to_thread(read_stream)
    return events
```

## 4. Thread Safety & Database Transaction Deadlocks

### Thread Safety of SDK Clients
- **Safe Concurrent HTTP Requests**: The `Lemma` client encapsulates an `AuthenticatedClient`, which initializes and caches a single `httpx.Client`. The synchronous HTTP client in `httpx` is thread-safe for concurrent HTTP operations (sharing a thread-safe connection pool).
- **Client Sharing**: Multiple threads (e.g. FastAPI worker threads) can safely share a single instance of `Lemma` or `Pod` to execute concurrent API requests.

### Database Isolation and Deadlocks
- **Request-Scoped Backend Transactions**: Every HTTP request made by the client corresponds to a distinct network request on the backend. The backend handles database transactions on a **request-scoped basis** (typically within a dedicated unit of work lifecycle). Concurrently executed SDK calls will never share a database transaction or connection.
- **Deadlocks**: Because transactions are request-scoped, deadlock risks on the database are identical to any concurrent database application:
  - Concurrent writes targeting the same database rows or modifying identical schema attributes concurrently (e.g. running `table_column_add` in parallel on the same table) can result in lock conflicts.
  - **Mitigation via RLS**: Row-Level Security (RLS) is enabled by default on new Lemma tables. This scopes operations to the current authenticated `owner_user_id`, reducing cross-user locking conflicts and transaction deadlocks.
